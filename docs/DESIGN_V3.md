## Problem Statement

You have two webradios running on the same Whatbox: **v1** (Node/Express + React + Vite, the one currently serving `radio.nicemouth.box.ca`) and **v2** (Go bridge + Next.js 16 + Postgres + Redis + Meilisearch, mid-rewrite, not yet user-facing). Neither is broken from a listener's perspective. Both are painful from yours.

v1 works but its Plex coupling is duct tape: a Python cron script runs every 6 hours, copies matching files from Plex's library into `~/.local/share/mpd/<stage>/music/`, rewrites JSON sidecars, reads the Plex token out of `Preferences.xml`. When it breaks, debugging it means digging through three layers of indirection.

v2 was started to solve v1's issues but the scope kept growing. It now depends on Postgres, Redis, Meilisearch, a Go bridge, and a Next.js 16 frontend. Redis and Meilisearch are declared in `podman-compose.yml` but aren't even started. Analytics features are the goal but Step 5 is the current block. The ops surface grew while the product didn't.

The restart isn't about features listeners are missing. It's about consolidation: ONE codebase, cleaner Plex coupling, modern-feeling stack without v2's scope explosion. Analytics (what v2 was chasing: artists, labels, albums, play history, wrapped) comes back progressively in v3.1+, after the foundation is solid.

## Product Truth

**The radio is a sibling of the DJ rig.** Your Plex playlists aren't curated for the radio, they're curated for your DJ sets — a tool auto-syncs them into Rekordbox, which you use when DJing at Pavoia. The radio plays the exact same tracks a Pavoia dancer might hear in person that weekend. That's not "my music collection on shuffle." That's "you're listening to what's live in my crate right now." Plex is the single source of truth, two downstream consumers (radio + Rekordbox) both read from it, no duplication of curation work.

**The three-tier curation funnel.** Not every track in your Plex is radio-ready. There's a funnel:

```
Listen (34,562 tracks)      ← discovery pile, tracks arrive here
        │
        │ "I liked it"
        ▼
Again (2,864 tracks)        ← candidate queue, second-pass evaluation
        │
        │ "I loved it again"
        ▼
Pavoia stages (1k–2k each)  ← curated rotation, plays on the radio
```

The radio only reads from tier 3. The Listen and Again playlists exist in Plex but are invisible to v3 — they're your private workflow. A track you heard on Fontanna today was promoted *twice* to get there: once from the discovery pile to Again, then from Again to its venue home. That's what "curated" actually means here.

**The funnel resets periodically.** Each curation iteration, Listen and Again get emptied and rebuild from scratch. Tier 3 (the stage playlists) is the only permanent state. **Implication for v3**: the engine never needs to read tiers 1 and 2 at all. **Implication for v3.1+ analytics**: if you ever want promotion-velocity stats ("Track X lived in Again for 40 days before being promoted to Bermuda Night"), v3 would need to snapshot tiers 1 and 2 itself before each reset — Plex won't have that history. File that under "later, if you care."

## What Makes This Cool

**Zero-copy Plex coupling.** The v1 "sync" deletes itself. Plex is the source of truth, the engine reads playlists via API and audio bytes direct from filesystem. Add a track to a Plex playlist (as you already do while building your DJ sets), it's in rotation on the radio within 60 seconds. No cron, no Python, no JSON rewriting. No separate "radio curation" workflow — the DJ workflow IS the radio workflow.

**HLS-native streaming.** The same technology Spotify, Apple Music, and YouTube use. Listeners get genuine resilience to network blips (client buffers 3-4 segments), native mobile support (iOS/Safari plays it out of the box), CDN-ready, and deployment independence from Icecast-style persistent connections.

**Modern TypeScript across the whole codebase.** One language, shared types between engine and UI. Open any file, you know what's going on.

**Same box, no moving parts bloat.** Whatbox stays. Plex stays. MPD dies. The 9 MPD daemons, 9 configs, 9 ports, self-built MPD binary after the Gentoo libfmt debacle — all gone, replaced by one engine process.

## Constraints

- **Solo maintainer.** Must be fun on a Saturday and readable 6 months from now.
- **Runs on Whatbox** (`orange.whatbox.ca`). No new hosts. Reuses the `*.nicemouth.box.ca` subdomain TLS that Whatbox provides.
- **Plex stays.** Plex manages the music library, playlists are defined in Plex, v3 never tries to replace that role.
- **No MPD in v3.** All 9 MPD instances shut down at cutover.
- **No Postgres / Redis / Meilisearch in v3.0.** Those come back only if a v3.1+ feature actually needs them.
- **Built with Claude Code.** Stack choices bias toward what Claude writes well today.
- **Listeners must not notice the cutover.** v1 stays up until v3 is verified.

## Premises (agreed)

1. v3.0 ships JUST the listener experience — player UI, audio engine, Plex coupling, deployment. NO analytics, NO Postgres, NO listener counting, NO history persistence.
2. v2's existing Postgres data (plays history, enrichments) gets abandoned at cutover — clean slate.
3. v3 deploys to the same Whatbox (`orange.whatbox.ca`), but ditches rsync-and-kill-by-PID in favor of a modern pipeline: GitHub Actions → graceful process reload via systemd or pm2.
4. Cutover is parallel-subdomain: v3 runs at `v3.nicemouth.box.ca` until verified, then `radio.*` / `pavoia.*` get repointed, v1 and v2 get decommissioned.

## Approaches Considered

### Approach A: Two processes, monorepo (RECOMMENDED)
```
pavoia-webradio-v3/
├── apps/
│   ├── engine/       Node daemon: Plex client, ffmpeg manager, HLS writer, WebSocket hub
│   └── web/          Hono server + Vite SPA build: API, static serving, WebSocket proxy
├── packages/
│   └── shared/       TypeScript types, zod schemas, stage definitions
├── deploy/           systemd unit files, deploy.sh, GitHub Actions workflow
└── docs/             README, CLAUDE.md
```
- **Summary:** engine and web are separate Node processes, sharing types via `packages/shared`. engine writes HLS segments to `/var/tmp/radio-hls/<stage>/` (tmpfs), web serves them statically and proxies WebSocket events.
- **Effort:** M
- **Risk:** Low
- **Pros:**
  - Web restart (deploy) doesn't interrupt audio
  - Clean failure isolation — engine crash doesn't take down the player UI (listeners just see a "reconnecting" state)
  - Natural extension point for v3.1+ analytics (add `apps/analytics` without touching engine)
- **Cons:**
  - Two systemd units, two deploy targets, slightly more to reason about than a single process
  - IPC via WebSocket adds a tiny bit of plumbing
- **Reuses:** v1's stream metadata structure and stage-ordering logic, v2's WebSocket hub pattern, v1's per-stream gradient/accent visual system (streamMeta.js is genuinely good and ports cleanly)

### Approach B: One process, workers per stage
- **Summary:** single Node process, 9 `worker_threads` for ffmpeg management, main thread serves HTTP + HLS + WebSocket.
- **Effort:** M-L
- **Risk:** Med
- **Pros:** one process to deploy, modern Node concurrency, one `pm2 restart`
- **Cons:** web restart interrupts all 9 streams; worker_threads IPC is more painful than WebSocket; a misbehaving worker can still GC-pause the main thread
- **Reuses:** same as A

### Approach C: Single everything
- **Summary:** one Node process, engine + HTTP + static + HLS writing all in the same event loop.
- **Effort:** S
- **Risk:** Med-High
- **Pros:** simplest possible thing, ship in a weekend
- **Cons:** no failure isolation, engine CPU spike blocks HTTP serving, tight coupling means future analytics work touches everything
- **Reuses:** same as A, but less of it

## Recommended Approach

**Approach A** (two processes, monorepo).

Rationale: the biggest risk in a foundation rebuild is that you ship something that *feels* clean initially but traps you when it grows. Approach C ships fastest but guarantees you'll be rewriting again in 6 months when the first analytics feature arrives and you need to not disrupt audio while deploying it. Approach A costs maybe a day more of initial setup and buys clean extension points forever. Approach B is a worse version of A — same number of hard problems, less separation of concerns.

The pattern matches what actually worked in v2: engine (MPD watcher + enrichment) and bridge (API + web) were separate processes, and that part of v2 is stable. The part that wasn't stable was the exploding dependency tree below them. v3 keeps the process separation, drops the DB/cache/search stack until needed.

## v3.0 Feature Scope (the "best of the best" MVP)

User's bar (2026-04-21): not just a v1 replacement. v3.0 ships with everything that makes the radio genuinely *better* than v1 on day one, minus v2's configurable-everything complexity. Six substantive pillars:

### 1. Listening
- **11 stages** (10 Plex-backed + Bus mystery). See "Plex Audit Results" below for the locked mapping.
- **HLS streaming.** One `.m3u8` per stage, 3-sec AAC 128 segments, rolling window of ~6.
- **Plex → stage binding.** Engine reads each stage's Plex playlist every 60s. Adds/removes detected, queue updates atomically at next track boundary.
- **Client-side crossfade** on stage switch (fixed curve, ~1.2s ramp, two `<audio>` alternation — port v1's pattern).
- **Full player UX.** Browse & Switch, now-playing strip when exploring, "was playing" toast, hover preview, per-stage accents/gradients (port `streamMeta.js`), descriptions from Plex `summary` field.
- **Controls.** Play/pause, volume + mute (persisted), keyboard shortcuts (Space, ←→, ↑↓, M, 1–9+0, Enter, Esc), swipe navigation on mobile.
- **Mobile UX.** 64px mini player, full-screen expand via swipe up, streams drawer, bottom tab bar (Home / History / Discovery / Artists / More), haptic feedback on switch.
- **Media Session API** for lockscreen controls, **PWA** install + offline shell, **background audio**.

### 2. Knowing what stage you're on
- Persistent visual identity per stage (icon + gradient + accent color).
- Stage title and description pulled live from Plex playlist metadata (edit in Plex, radio updates within 60s).
- Live status indicator (pulsing dot when a stage's HLS manifest is updating).
- Bus mystery card (port v1's full-screen overlay verbatim).

### 3. Artist / album / label info
- **Artist pages** (dedicated route per artist). Bio, photo, country with flag, genre tags, external links (Spotify, Bandcamp, SoundCloud, Resident Advisor, Wikipedia), library tracks grouped by stage, similar artists (clickable if in library), stage-presence bars ("appears on 3 stages: Fontanna, Palac Dance, Etage 0").
- **Album pages.** Cover, year, label, track list with in-library vs. full-album counts, total plays (once history accumulates).
- **Label pages.** Label name, artists under this label in the library, total tracks.
- **Artist drawer** (quick access from now-playing — port v1's pattern).
- **New artist badge** (🆕 first time an artist appears in any library event).
- **Enrichment pipeline** — two sources only: **Last.fm** (primary: bio, tags, similar, listener counts) + **MusicBrainz** (MBID, country, relationships). Discogs / Wikidata / TheAudioDB explicitly dropped. Enrichment runs as a background worker in the engine, cached in SQLite. First encounter triggers enrichment; subsequent requests hit cache.

### 4. Play history
- **Every track change logged** to SQLite: `(stage, track_id, artist, title, album, started_at)`. Zero Postgres.
- **Timeline view** (dedicated route `/history`). Day-grouped, reverse-chronological.
- **Filters**: stage, time range (presets + custom), artist type-ahead.
- **Stats sidebar**: today/week/month totals, top track.
- **CSV export** for curator.

### 5. Discovery feed (new additions per stage)
- **Engine observes Plex playlists over time.** Every 60s poll, diffs each stage's playlist, logs first-seen-in-stage events to SQLite. This is the ONLY way to capture additions — user's curation funnel (Listen/Again tiers) gets reset periodically, so Plex itself doesn't preserve this history.
- **Per-stage discovery feed** ("New in Palac Dance this week").
- **Global discovery feed** across all stages.
- **Filters**: stage, time range. Each addition row has: play, go to artist, download.

### 6. Live listener count
- **Per-stage count** of active HLS segment requesters (rolling window, e.g., "requested a segment in last 15s").
- **Total across stages.**
- **WebSocket push** to all connected clients on count change (debounced).
- **Milestone toasts** on round numbers: "🎉 100 live on Bermuda!"

### Plus: track download
- Per track (from now-playing, history rows, discovery feed, artist pages). Serve the source audio file from Plex's library with proper `Content-Disposition`. User override — was DROP, is now MVP.

### Plus: share & deep-links
- Native share (stations, stages, tracks, artists). Open Graph preview cards for rich link previews in iMessage/Twitter/Slack.

### Plus: accessibility + a11y
- ARIA labels, visible focus rings, screen reader announcements on track change (aria-live), WCAG AA contrast, `prefers-reduced-motion` respected, reduce-motion toggle.

### Plus: deploy & ops
- GitHub Actions → typecheck → test → build → rsync → `systemctl --user reload radio-engine radio-web`. Zero-drop engine reload (keep old ffmpeg running during web switch). Parallel subdomain cutover (`v3.nicemouth.box.ca`).

### Explicitly OUT of v3.0

- **Admin panel** (user drop — no UI, `stages.ts` is source, `ssh` is ops)
- **Queue display / up-next** (user drop — radio magic is surprise)
- **Camelot key wheel / harmonic mixing** (user drop — DJ tool, not listener feature)
- **Configurable-everything** (5 crossfade curves, 3 themes, 3 layouts, font/shape settings — user drop, pick-one-great-default)
- **Wrapped** (v3.1 — needs year of data)
- **Curator dashboard** (v3.2 — digging calendar, streaks, insights)
- **Advanced analytics** (v3.2 — decade breakdown, genre evolution, geo artist map, heatmaps)
- **Artist graph visualization** (v3.3)
- **Search** (v3.2 — wait until there's enough indexed to matter)
- **Postgres / Redis / Meilisearch** (never, unless SQLite proves insufficient)
- **Listener auth / user accounts** (never)

## Cross-Stack Details

### Engine (apps/engine)

- **Runtime:** Node 22 LTS, TypeScript strict mode.
- **HTTP:** Hono (for the internal event API that web talks to).
- **Audio:** `child_process.spawn('ffmpeg', […])` per stage. One ffmpeg process per stage, reads audio files from Plex library path, encodes to AAC, writes HLS segments.
- **Plex client:** raw `fetch` to `http://127.0.0.1:32400/playlists/<id>/items` with `X-Plex-Token` header. Token loaded from env var (`PLEX_TOKEN`), not parsed from `Preferences.xml` anymore.
- **WebSocket hub:** `ws` package. Broadcasts `{type: "track_changed", stage, track}` on ffmpeg stdout-parsed ICY events or on track-boundary detection via HLS segment output.
- **State:** in-memory per stage: `{queue: Track[], current: Track, position: number}`. No DB.
- **Queue refresh:** every 60s (or on webhook if we wire that up later), engine re-reads Plex playlist, diffs against its queue, updates atomically at next track boundary (never mid-track).
- **Stage config:** `packages/shared/stages.ts` — the 9 stage IDs mapped to Plex playlist IDs + visual metadata (icon, title, desc, gradient, accent color). Single source of truth.
- **systemd unit:** `radio-engine.service`, user-level, Restart=always, graceful SIGTERM handling (drains ffmpeg cleanly, finishes current segment).

### Web (apps/web)

- **Runtime:** Node 22 LTS, TypeScript strict mode.
- **Framework:** Hono (HTTP + static file serving + WebSocket proxy).
- **Frontend:** Vite + React 19 + TanStack Router + TanStack Query + Tailwind 4 + TypeScript. Built into `dist/` and served statically by Hono.
- **Endpoints:**
  - `GET /api/stages` — list 9 stages with metadata
  - `GET /api/stages/:id/now` — current track (polling fallback)
  - `GET /api/artists/:name` — artist data (from Plex API, proxied + cached)
  - `GET /hls/:stage/index.m3u8` — HLS playlist (static, served from tmpfs)
  - `GET /hls/:stage/segment-*.ts` — HLS segments (static, cache-control: short)
  - `WS /ws` — WebSocket for real-time now-playing events
- **Auth:** none (public read-only radio).
- **systemd unit:** `radio-web.service`, user-level, Restart=always.

### Shared (packages/shared)

- **Stage definitions:** canonical list of 9 stages with id/name/icon/desc/gradient/accent/plex-playlist-id.
- **Types:** `Track`, `Stage`, `NowPlaying`, `TrackChangedEvent`, `ArtistCard`.
- **Zod schemas:** runtime validation on the WebSocket boundary and Plex API responses.

### Filesystem layout on Whatbox

```
~/pavoia-webradio-v3/              # git checkout
  ├── apps/engine/dist/             # built
  ├── apps/web/dist/                # built (SPA assets)
  └── deploy/radio-*.service        # systemd units, symlinked into ~/.config/systemd/user/

/var/tmp/radio-hls/<stage>/         # tmpfs, HLS segments + m3u8
  ├── index.m3u8
  ├── segment-0001.ts
  └── segment-0002.ts

~/.config/systemd/user/             # user systemd
  ├── radio-engine.service
  └── radio-web.service

~/.config/radio/env                 # secrets (PLEX_TOKEN, etc.)
```

The old MPD directories (`~/.config/mpd/*`, `~/.local/share/mpd/*`) are left in place during v3.0 testing and decommissioned at cutover. The old user-space nginx (`~/.config/nginx/`) can stay — it's routing v1/v2 and gets retired at cutover too.

## Success Criteria

The design is successful when:

1. **Cutover is invisible to listeners.** Someone listening to `radio.nicemouth.box.ca` at the moment of cutover doesn't notice more than a 3-second rebuffer.
2. **Add-a-track-to-Plex-playlist → in rotation within 90 seconds**, consistently, for all 9 stages. No script to kick, no cron to wait for.
3. **You can explain the entire v3 architecture from memory in 6 months.** That's the maintainability gate.
4. **Deploying a UI change from laptop to live is one `git push`** and takes < 90 seconds end to end.
5. **Zero MPD processes running on Whatbox post-cutover.** v1 and v2 fully decommissioned, disk reclaimed.
6. **Mobile experience on an iPhone during a walk through a subway tunnel doesn't drop the stream** (HLS buffer absorbs the gap).

## Open Questions

1. **Exact HLS segment length.** 3 seconds is a good default for radio latency vs. buffer-resilience. Can tune after first listen.
2. **What happens when a Plex playlist is empty?** Engine behavior: skip the stage (empty `.m3u8`)? Fall back to a looped silence marker? Probably: emit a short "this stage is being curated" audio loop. Decide when we see it happen.
3. **ffmpeg flags for ICY-free continuous output.** HLS output from ffmpeg is standard (`-f hls`), but track-boundary detection for now-playing needs either stderr parsing of ffmpeg's `size=... time=...` lines or a per-track ffmpeg invocation (cleaner but more process churn). Start with the per-track approach, measure.
4. **Does the v1 "bus" subdomain 502 issue need to be fixed before cutover?** No — v3 supersedes both v1 port setups. Just don't repeat the split-port bug.
5. **Public read-only listener counting without persistence** — is the count per-stage or global? Per-stage is more useful and trivial to maintain in-memory.
6. **Where do stage icons + gradient colors live?** Hybrid: icon/gradientFrom/gradientVia/gradientTo/accentColor stay in `packages/shared/stages.ts` (visual identity, rarely changes). Title + description come from Plex dynamically (curation flow, user edits in Plex). v1's `streamMeta.js` visual system ports cleanly as the static half.
